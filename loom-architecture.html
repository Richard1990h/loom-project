LOOM

Looped Operations Over Memory — a small model that loops its compute and looks up its knowledge.

  • 0.4B shared-weight core
  • knowledge in storage, not weights
  • depth 2–24 · learned per token

STATUS: BLUEPRINT · NO EMPIRICAL RESULTS · HYPOTHESES TO FALSIFY

The bet

A large language model is mostly a warehouse with a small workshop attached. LOOM proposes we stop shipping the warehouse with every token.

Most parameters in a dense model sit in feed-forward layers acting as key–value stores for facts — and every one of those weights must stream through the processor for every token, whether the token needs them or not. On consumer hardware, that memory traffic, not arithmetic, is the bill. LOOM splits the two jobs: a tiny reasoning core small enough to stay cache-resident, run in a loop as many times as a token deserves, reading facts from a huge, cheap, editable latent memory that lives on ordinary storage. Depth becomes a dial. Knowledge becomes a file.

Mechanisms

Four parts, named for the machine that invented programmability

  • Shuttle · the recursive core

    Depth is a dial, not a purchase

    One stack of four shared transformer blocks (~0.4B params) iterated 2–24 times per token group, with a learned halting head trained against an explicit compute price. The weights fit in on-chip cache and stay there — each extra "layer" costs arithmetic but zero extra weight traffic. Like a shuttle, it crosses the same shed again and again; the cloth gets deeper, the machine doesn't get bigger.

  • Warp beam · externalized knowledge

    Facts live in storage, not in matmuls

    A product-key latent memory: billions of (key, value) vector pairs written during training, held int8 in RAM and NVMe, retrieved in O(√N) with two-level keys — top-8 reads per loop. This is not text RAG bolted on after; the values are learned latents, gradient-shaped to be exactly what the core wants to consume. FFNs shrink to thin skill layers. Side effect: knowledge becomes editable and auditable, entry by entry, with no retraining.

  • Heddles · thought registers

    Chain-of-thought without the tokens

    Thirty-two persistent latent slots read and written by cross-attention on every loop, surviving across iterations and across positions in a segment — the same scratchpad serves depth and time. The model drafts a plan in vector space before it spends a single output token, the way heddles hold the pattern between passes of the shuttle. Trained by distilling a teacher's written rationales into register trajectories.

  • Jacquard head · context hypernetwork

    The model reprograms itself per problem

    A ~12M-parameter controller reads the register summary and emits rank-8 weight deltas for the shared blocks — refreshed every loop, so iteration 3 can run different effective weights than iteration 1. One physical core, many virtual specialists. In 1804 Jacquard's punch-card head made a loom the first programmable machine; here the punch card advances once per pass.

The names aren't decoration — each mapping is doing work. The shuttle really does loop; the warp beam really is where all the thread is stored; heddles really do hold state between passes; and the Jacquard attachment really was a program that reconfigured the machine per pattern. The whole architecture is a two-century-old idea returned home: a small machine, richly re-instructed, beats a big machine run once.

Why the small platform bangs bigger

Pay for thinking only where thinking happens

On phones and laptops, token latency is dominated by streaming weights from memory, not by math. A cache-resident core makes that cost flat no matter how deep you loop — and the halting head only loops where the token is hard. Drag the dial:

Back-of-envelope arithmetic (2·params per forward pass, int8 weights), for intuition only — no KV-cache subtleties, no benchmarks. The point survives the sloppiness: even at maximum depth, LOOM's compute reaches roughly dense-7B parity while its weight traffic stays two orders of magnitude smaller — and the average token exits after a handful of loops.

The second economy is knowledge. Model capacity bought as HBM-resident weights costs orders of magnitude more per gigabyte than capacity bought as NVMe-resident memory entries. LOOM moves the fact warehouse onto the cheap curve — and because entries are discrete, a wrong fact is fixed by writing one row, not by finetuning a network.

Honesty ledger

What's inherited, what's actually new

Every thread here has been spun somewhere in the literature. Claiming otherwise would be false. The bet is in the knots — the way they're tied together, and one deliberately hostile training objective.

Inherited threads

  • Weight-tied recursion — Universal Transformers '18; Mixture-of-Recursions '25
  • Learned halting — Adaptive Computation Time '16; PonderNet '21
  • Product-key memory layers — Lample et al. '19; scaled memory layers '24–'25
  • Retrieval as capacity — kNN-LM '20; RETRO '22; Memorizing Transformers '22
  • Latent-space reasoning — Coconut '24
  • Hypernetworks & low-rank deltas — Ha et al. '16; LoRA '21

New knots — the bets

  • Knowledge starving. Train with FFNs deliberately too thin plus a penalty on closed-book recall, so facts have nowhere to live except memory. Memory as the primary store, not a bolt-on — the untested core of the whole design.
  • Modulation inside the loop. Hypernet deltas refreshed per iteration, so recursion isn't running the same function k times but a short learned program of k related functions.
  • Halting priced in hardware terms. The ponder cost tied to a real budget (joules, ms), making depth a deployment knob you can tune after training.
  • One scratchpad for depth and time. Registers shared across loop iterations and sequence positions, coupling latent planning to adaptive depth.

Open problems a serious attempt would hit first

  • Gradient stability through 24 weight-tied iterations with per-loop deltas — recursion training is notoriously touchy.
  • The memory write policy: what earns an entry, when entries merge, how to stop retrieval collapse onto a few hot keys.
  • NVMe tail latency turning into token-time jitter; hot-set caching has to actually work.
  • Evaluation hygiene: a writable memory can memorize the test set unless provenance is tracked per entry.
  • Whether "knowledge starving" harms fluent composition — some facts may need to live in weights to be usable at speed.
